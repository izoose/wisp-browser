Where it goes
The process explosion

Chromium spawns a process per site, plus GPU, network and utility helpers. For eight tabs, Chrome fanned out to 59 processes. Wisp's slept state runs in 8. Every live renderer is memory you're paying for.

The mechanism
How a tab lets go of its memory

This is the whole trick — and where the 2,204 → 711 MB drop comes from.

Full speed, all its memory — exactly like any browser.

How it was measured
Method
Tabs
YouTube, Reddit, Wikipedia, GitHub Trending, CNN, Amazon, weather.com, ESPN — identical for all three.
Isolation
Each browser ran on its own throwaway profile (fresh — no extensions, sessions or open tabs). Wisp ran with its single-instance lock bypassed so it didn't merge into the everyday copy.
Metric
Sum of the working set (physical RAM) of every process in that browser's tree, collected 65 s after launch.
Wisp ×2
Measured right after load (all live) and again after its sleep/discard timers ran — the realistic state.

Being honest
Caveats
  • Working set counts some shared memory per process, so absolute totals run a little high — but all three are Chromium, so the comparison is apples-to-apples.
  • Single run on one machine; RAM fluctuates a few percent between runs. Treat the numbers as directional — the 3–7× ratios are large and stable enough to matter.
  • Any Chromium browser uses similar RAM per live tab. Wisp doesn't beat physics on an active page — it wins by not keeping idle tabs alive.
  • Chrome's high figure reflects its default full site-isolation and helper processes; a machine with fewer cores may show fewer.
